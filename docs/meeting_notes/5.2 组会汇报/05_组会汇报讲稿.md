# 05 组会汇报讲稿

## 开场：问题和定位

今天我想汇报 KGTraceVis 最近整理后的系统定位和下一步计划。这个项目不是要再训练一个异常检测模型，而是处理异常检测之后的问题：图像、时间序列和日志模型都会给出异常输出，但这些输出格式不同、语义不同，也很难直接追溯到知识、证据和候选根因。

所以 KGTraceVis 的定位是：把异构异常输出转成统一 Evidence JSON，再通过 source-constrained KG（来源约束知识图谱）做 entity linking（实体链接）、consistency checking（一致性检查）、correction candidate generation（修正候选生成）和 RCA（Root Cause Analysis，根因分析）候选路径排序。最后在可视分析界面里展示 raw evidence、normalized evidence、linked entities、correction candidates、top-k paths 和 source edge provenance。

## 总体路线

现在的路线分四步。

第一步是准备数据和 reference 边界。每个样本都应该只包含可观察异常证据，比如 object、anomaly_type、location、morphology、variables、variable_contributions 和 log_events。Root-cause reference 要单独管理，并且区分 native ground truth、official fault type、literature-supported、manual plausible、llm_candidate 和 demo_synthetic。

第二步是 source-constrained KG construction。KG 不是大而全的工业知识库，而是围绕 evidence normalization、consistency checking、correction、path ranking 和 visual analytics 的任务图谱。每条 edge 都必须有 source、evidence、confidence、review_status 和 feedback counters。

第三步是 dataset-specific adapters。这里最重要的边界是：adapter 只产出 observed evidence，不产出 root cause，不产出 top-k paths，也不预填 `kg_analysis`。

第四步是 KGTracePipeline runtime reasoning。输入 Evidence JSON 后，pipeline 运行时完成 linking、consistency、correction 和 candidate/plausible RCA path ranking。

## 数据集分工

三个数据集的角色不能混在一起。

MVTec/DS-MVTec 主要做视觉 evidence normalization、morphology/location consistency、噪声修正和 plausible explanation demo。这里必须强调：MVTec 不提供 verified factory RCA ground truth，所以它的 path 只能说是 curated plausible references 下的 candidate explanation。

TEP 是下一阶段最适合作为主 RCA/path-ranking 量化的方向。它有过程变量、fault type、process unit 和 fault propagation 语义，可以做 variable-unit-fault-root-cause linking、top-k path hit、MRR 和 ablation。

Wafer 目前更适合做 image/log/process traceability case study。如果没有私有 process logs、lot history、tool/chamber records 和专家 RCA labels，就不应该声称它能做大规模 RCA accuracy。

## Adapter 如何生成 Evidence JSON

以 MVTec 为例，PatchCore 或 EfficientAD 可以输出 anomaly map 和 pred_score，DefectSpectrum 或 DS-MVTec 提供 defect label、caption 和 mask，mask geometry 可以计算 location、morphology 和 severity。可选 VLM 只能产出弱 observation，比如 caption、candidate defect type、location/morphology hint，并且要标成 low confidence、needs review。

以 TEP 为例，PCA、DPCA、RBC contribution 或 autoencoder residual 可以产生 abnormal variables、contribution scores 和 temporal pattern。Fault number 不应进入 adapter 输出，而应放在 reference/evaluation 文件中。

以 wafer 为例，CNN/ResNet/ViT 可以产出 spatial pattern 和 confidence，空间 descriptor 产出 location、morphology、severity。如果没有真实 log，只能用 demo_synthetic log，不能伪装成真实多模态 RCA。

这条边界可以概括成一句话：model/adapter 产出 observed evidence，KGTracePipeline 产出 candidate explanation，human review 再决定接受或拒绝。

## 当前 Pipeline 是什么水平

当前仓库已有可运行的 v0 `KGTracePipeline`。它会调用 entity linker、consistency checker、correction generator 和 path ranker。Path ranking 使用的公式是：

```text
Score(P) = alpha * Conf(P) + beta * EvidenceMatch(P) - gamma * Length(P)
```

每条 path 有 stable path_id、node sequence、relation sequence、score、supporting evidence 和 source edges。这已经可以支撑组会 demo，尤其是 what-if 编辑后重新计算，证明不是静态 JSON 展示。

但要诚实地说，当前实现还是 deterministic entity linking + KG constraint checking + graph search ranking baseline。它不是 causal discovery，也不是成熟工业 RCA 系统。下一步要增强的是 TEP 量化、noise/correction protocol、annotation taxonomy、provenance metadata 和 relation-weighted ablation。

## 上中下三策

下策是把系统包装成自动工业 RCA，甚至把 MVTec 的人工 plausible reference 当真实根因标签。这条路风险最大，不建议。

上策是做动态因果/概率 KG，比如 Online Evidence Graph（在线证据图）、Bayesian Network、causal graph 或 temporal GNN，再结合专家反馈闭环。这是长期方向，但对数据、标签和实现质量要求太高，不适合作为当前组会承诺。

中策是最稳的：source-constrained KG baseline + TEP 主量化。也就是把 KGTraceVis 定位为统一 evidence schema、来源约束 KG、runtime pipeline 和可视化审阅系统；MVTec 做视觉 evidence 和 correction demo，TEP 做主 RCA/path-ranking evaluation，wafer 做 traceability case study。

## 结束：下一步工作

下一步我建议优先做四件事。

第一，补强 TEP 的 variable-unit-fault-root-cause mapping 和 reference 文件，作为主 path-ranking 评估。

第二，固定 noise experiment protocol，包括 noise type、level、seed、clean reference 和 reporting format。

第三，继续保证 adapter-vs-KG boundary：example evidence 的 `kg_analysis` 初始为空，root cause 不作为输入字段。

第四，在 UI 和文档里持续展示 source edge provenance、stable correction/path IDs 和 human feedback hooks，保证系统是可审阅、可追溯的研究原型。

最终对外叙事可以是：KGTraceVis 解决的不是“自动给出真实根因”，而是把异常检测输出变成统一、可验证、可追溯的 evidence，并用来源约束知识图谱在运行时生成可解释的修正候选和候选 RCA 路径。
