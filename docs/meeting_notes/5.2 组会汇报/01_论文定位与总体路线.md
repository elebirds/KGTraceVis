# 01 论文定位与总体路线

## 推荐论文定位

建议将 KGTraceVis 定位为：

> KGTraceVis: Source-Constrained Knowledge Graph Reasoning over Unified Industrial Anomaly Evidence for Explainable Traceability

中文表达：

> KGTraceVis 是一个面向工业异常输出的知识约束 evidence analysis 与 traceability pipeline。它将图像、时间序列和日志异常输出转换为统一 Evidence JSON，再通过 source-constrained KG（来源约束知识图谱）执行可解释的实体链接、一致性检查、噪声 evidence 修正和候选根因路径排序，并在可视分析界面中暴露 provenance（来源证据）与 human feedback（人工反馈）入口。

这个定位比“自动发现真实工业根因”更稳妥。它强调异常检测之后的证据组织、知识约束推理和可追溯审阅，避免因为公开数据集缺少真实 RCA（Root Cause Analysis，根因分析）标签而被质疑。

## 四阶段路线

### 1. 数据集准备与 reference 边界

为 MVTec/DS-MVTec、TEP、wafer 准备可进入统一 schema 的样本。样本应包含可观察字段，例如 object、anomaly_type、location、morphology、variables、variable_contributions、log_events、timestamp 和 raw evidence。

Root-cause reference 必须分层：

- native ground truth：数据集原生标签，例如 mask 或明确 fault id。
- official fault type：官方 benchmark 定义的 fault 类型。
- literature-supported：论文或技术文档支持的 cause/path。
- manual plausible：项目人工整理的合理参考，不等同真实工厂标签。
- llm_candidate：LLM 从给定来源中抽取的候选，必须等待审阅。
- demo_synthetic：为了展示冲突、噪声或 UI 构造的样例。

### 2. Source-constrained semi-automatic KG construction

收集数据集说明、官方表格、论文、项目 note 等来源材料，登记到 source registry 或 docs/sources。LLM 可以抽取 candidate triples，但只能从提供来源中抽取，不能补外部常识。每条 KG edge 必须包含 source、evidence、confidence、review_status 和 feedback counters。

QA 至少检查 required columns、节点引用、confidence/weight、review_status、重复边、孤立节点 warning 和 feedback counters。`review_status=reviewed` 的边不能被自动抽取结果覆盖。

### 3. Dataset-specific adapters 生成统一 Evidence JSON

每个 adapter 只负责把 raw data、模型输出或人工 demo record 转换为 observed anomaly evidence。`kg_analysis` 初始为空，root cause、top-k paths、ranked causes 不进入 adapter 输出。

这一步的贡献是把异构异常输出变成统一、可验证、可追溯的输入合同，而不是提前把答案写进 JSON。

### 4. KGTracePipeline runtime reasoning

`KGTracePipeline.analyze(evidence)` 在运行时完成：

1. entity linking（实体链接）：把 evidence mentions 链接到 KG 节点。
2. consistency checking（一致性检查）：检查异常类型、形态、位置、变量、日志事件之间是否满足 KG 约束。
3. correction candidate generation（修正候选生成）：在字段冲突时生成有 source edge 支撑的候选。
4. relation-weighted RCA path ranking（关系加权根因路径排序）：计算 candidate/plausible paths，并输出 stable path IDs、node/relation sequence、score、supporting evidence 和 source edges。

输出是候选解释，不是最终事实判定。

## 数据集角色分工

| 数据集 | 推荐角色 | 可评估内容 | 不应声称 |
| --- | --- | --- | --- |
| MVTec / DS-MVTec | 视觉异常 evidence normalization、morphology/location consistency、噪声修正和 plausible explanation demo | schema validity、entity linking、字段一致性、morphology/location correction、against curated plausible references 的 path hit | 原生真实工厂 RCA、verified process root cause |
| TEP | 主 RCA/path-ranking 量化场景 | variable-unit-fault-root-cause linking、top-k path hit、MRR、relation-weighted ranking ablation | 真实工厂生产事故；应说明是经典仿真 benchmark |
| Wafer | 多模态 image/log/process traceability case study | image/log evidence consistency、source edge provenance、case-level path plausibility、专家反馈或人工审阅记录 | 在缺少可靠标签时做大规模 RCA accuracy claim |

## 贡献边界

可以主张：

- 统一 anomaly evidence schema 让多源异常输出可验证、可比较、可追溯。
- Source-constrained KG 让 evidence correction 和 candidate path ranking 有来源约束。
- Runtime pipeline 让系统不是静态展示；what-if 编辑后可以重新链接、检查和排序。
- Stable IDs 和 source edges 让 correction、path、entity linking 和 KG edge 可被人工审阅。

不应主张：

- 系统已经自动发现真实工厂根因。
- MVTec/DS-MVTec 原生提供 factory RCA ground truth。
- Manual plausible reference 等同 verified ground truth。
- LLM/VLM 输出可以直接作为工业事实。

## 下一阶段最稳路线

短期用 MVTec 展示视觉 evidence 和 correction，用 TEP 做最关键的定量 path-ranking，用 wafer 展示多模态 traceability。论文主表优先放 TEP 和 noise/correction 结果；MVTec 和 wafer 更多承担 demo、case study 和边界清晰的辅助实验。
